# X-Check — Requirements

**Project:** X-Check (also read as "Cross-Check") — a personalized misinformation-intervention prototype
**Team:** PSIS SS2026, Group B (Amin, Jie, Tolga, Sarah)
**Status:** Draft · Living document — updated as decisions are made at the weekly team meeting
**Last updated:** 2026-06-25

---

## 1. Project Overview

X-Check is a Chrome browser extension that detects potential misinformation in social-media
posts and delivers **personalized** fact-check interventions inline. It is demonstrated on an
X (Twitter)-style mockup website built by our team.

Unlike generic, one-size-fits-all fact-checking, X-Check adapts both **how strongly** it
intervenes and **how the message is worded** to each individual user, based on a short
personality assessment and the user's real-time browsing behaviour. The goal is to shift from
reactive, generic fact-checking toward proactive, adaptive intervention — meeting each user
where they are psychologically.

This document defines the project's goals, scope, architecture, requirements, and roadmap. It
is the shared reference for all team members; when in doubt about *what we are building and
why*, this document is the source of truth.

---

## 2. Goals & Non-Goals

### Goals

- Deliver a working end-to-end personalization loop: **questionnaire → personality profile →
  misinformation detection → risk score → escalating intervention → LLM-adapted wording**.
- Run the full loop live on the X mockup website.
- Keep all sensitive user data **on the user's device** (local-first, privacy by design).

### Non-Goals (this version)

- **Analytics dashboard** (risk score, topic diversity, weekly reports) — deferred to Roadmap.
- **Backend service / API-key proxy** — deferred to Roadmap; this version is front-end only.
- **Deployment on real social platforms** — we only target our own mockup website.
- **Empirical effectiveness study** — building the prototype, not running a user study, is the
  scope of this version.

---

## 3. Scope & Definition of Done

### In scope (MVP)

1. Onboarding questionnaire and local personality profiling.
2. Live misinformation-detection pipeline connected to real Gemini 2.5 Flash and Google
   Fact Check Tools API calls. *(Now live on `feat/extension-scaffold`; see §8.)*
3. Risk Score computed from real-time **and** historical behaviour, including a learning loop.
4. Three escalating intervention types (T1/T2/T3), selected by Risk Score.
5. LLM-generated intervention wording adapted to the user's personality.

### Definition of Done

This version is considered successful when, on the X mockup website, the extension runs the
full chain end-to-end: **detect misinformation in a post → compute a Risk Score → trigger the
matching intervention level (T1/T2/T3) → render intervention text generated by the LLM and
adapted to the user's personality.**

Demonstrable acceptance checks:

- [ ] A new user completes the questionnaire and a personality profile is generated and stored
      locally. *(Questionnaire + profiling framework implemented on `feat/extension-scaffold` — this check is now demonstrable.)*
- [ ] Browsing to a known false post flags it and triggers the intervention type that matches
      the user's current Risk Score level.
- [ ] For the same post, two different personality profiles produce **differently worded**
      interventions.

## 4. System Architecture & Tech Stack

### Architecture: front-end-only prototype (Option A)

All logic runs inside the browser extension and the mockup page. There is **no backend**.
User data (questionnaire answers, personality profile, Risk Score history) is stored locally in
`chrome.storage` and never uploaded to a server. This is a deliberate privacy choice (see §6).

```
1_docs/                 Documentation (this file, etc.)
2_browser-extension/    Chrome extension (Manifest V3) — detection, scoring, interventions
  ├─ src/pipeline/      Fact-check pipeline (TypeScript): pipeline, geminiClient,
  │                     googleClient, prompts, parser, types
  └─ tests/pipeline/    Jest tests
3_mockup-website/       X-style mockup (React/JSX → Babel-compiled), served as static files
                        by Python on http://localhost:8000
```

### Tech stack

| Layer | Technology |
|---|---|
| Extension | Chrome Extension, Manifest V3, TypeScript |
| Detection pipeline | TypeScript (`extract → search → align → verdict → explain`) |
| LLM | Google Gemini 2.5 Flash (REST API) |
| Fact databases | Google Fact Check Tools API |
| Mockup website | React/JSX compiled with Babel, served via Python static server |
| Tests | Jest + ts-jest |
| Local storage | `chrome.storage` |

### Data flow (high level)

`Post on mockup page` → content script extracts text → pipeline detects misinformation →
Risk Score (from local profile + behaviour) selects intervention level → LLM generates
personality-adapted wording → intervention injected into the page.

## 5. Functional Requirements

| ID | Requirement |
|---|---|
| **FR1 — Onboarding questionnaire** | On first use, the user completes a **mandatory** BFI-10 test (10 items, Big Five). **Optional**: one political-orientation question with answer options `left` / `right` / no-answer (skip). NFCC has been removed from the questionnaire (decision: 2026-06-25 team meeting). A finer-grained political scale (strong vs. slight lean) is a planned future enhancement; see §9. |
| **FR2 — Personality profiling** | Questionnaire answers are converted **locally** into a small set of abstract personality labels (e.g. `openness: high`, `conscientiousness: low`). Raw answers stay on device. |
| **FR3 — Misinformation detection** | For each browsed post, the pipeline extracts claims, searches fact databases, aligns them with the claim, produces a verdict, and prepares an explanation. |
| **FR4 — Risk Score & behaviour tracking** | A per-user Risk Score is computed from **real-time behaviour** (scroll speed, dwell time, interactions) and **historical behaviour** (past responses to interventions). The score rises when the user ignores/dismisses interventions, dwells briefly on warnings, or shares flagged content; it falls when the user reads expanded warnings, clicks through to trusted sources, or spends time on the intervention. Responses continuously update the score (learning loop). *Conceptual framework only; exact weights are an implementation detail.* **Planned:** the mockup website now supports un-liking posts; the logic side must later lower the Risk Score when a like is undone (e.g., an `UNLIKE_FLAGGED` event with negative weight). |
| **FR5 — Intervention selection** | **The Risk Score alone determines the intervention level**: low → **T1 Label**, medium → **T2 Justification (with sources)**, high → **T3 Interruption (full-screen)**. Personality does **not** change the level — only the content (FR6). *(Only Openness, Agreeableness, and Conscientiousness affect wording; Extraversion and Neuroticism have no documented effect on misinformation-label attitudes per research.)* |
| **FR6 — LLM content adaptation** | For the selected intervention type, the LLM generates wording adapted to the user's personality (Openness, Agreeableness, Conscientiousness) and optional political orientation. User requests are **ID-anonymised**, so personal identity is not exposed even though political-derived tone guidance is included. Only **abstract personality labels**, optional political tone guidance, and the misinformation text are sent to the LLM — never raw answers or identifying information. |

## 6. Non-Functional Requirements

### Privacy (a core design principle)

- **Local-first / data minimization:** raw questionnaire answers, personality profile, and Risk
  Score history are stored only in `chrome.storage` and never uploaded. Because nothing is sent
  to a cloud database, the whole class of server-side breach/misuse risks does not apply.
- **Minimal LLM exposure:** anonymized personality labels (Openness, Agreeableness,
  Conscientiousness), optional political-derived tone guidance, and the misinformation text are
  sent to the LLM to generate intervention wording. User requests are ID-anonymised, so sending
  political tone guidance does not expose personal identity.

### Security

- **Known limitation:** in this front-end-only prototype, the Gemini and Fact Check Tools API
  keys are shipped client-side and are therefore exposed. We acknowledge this explicitly. A
  production version **must** move these calls behind a backend proxy. This is an accepted
  prototype trade-off, not an oversight.

### Performance & cost

- Gemini 2.5 Flash (free tier) shows ~5–15 s latency per call (typical ~6.6 s) and is rate- and
  reliability-limited (≈1,500 requests/day, occasional 503s, no retry logic). These constrain
  real-time use; a paid model or caching may be needed before final delivery (see §9).

## 9. Open Questions & Risks

- **Questionnaire instrument:** NFCC has been dropped (2026-06-25 meeting); BFI-10 + optional political question is now the fixed design. The political-question answer options are currently binary (`left` / `right`); a finer-grained scale is deferred to a future version.
- **LLM budget & latency fallback:** the free Gemini tier is too slow/limited for smooth
  real-time use; a budget request for a faster paid model (e.g. Claude Haiku, a premium Gemini,
  or DeepSeek) is pending. We intend to keep the original live, per-post detection design; **if
  the paid model is still not fast/affordable enough, the fallback is to pre-classify the mockup
  posts** (cache verdicts) and reserve live LLM calls for the personality-adapted wording step.
- **Data leaving the device (to revisit with §6):** detection is not purely on-device — the
  pipeline sends post content and extracted claims to Google (Gemini and the Fact Check Tools
  API) for 3 of the 5 steps. This is a known gap against the "local-first" framing in §6; the
  exact data-egress boundary and how it is described still needs to be reconciled.
- **Client-side API keys:** exposed in this prototype (see §6); additionally, keys are currently
  passed as a URL query parameter (`?key=...`), so they can surface in browser history, devtools,
  and proxy logs. Both must be addressed (backend proxy + header-based auth) before any real
  deployment. *(Security topics above are parked for a later discussion.)*
- **Risk Score tuning:** the scoring framework is agreed, but concrete weights and thresholds
  still need to be set and tuned during implementation.
- **UNLIKE_FLAGGED event (planned):** the mockup website now supports un-liking posts; the Risk
  Score logic must later lower the score when a like is undone (e.g., an `UNLIKE_FLAGGED` event
  with negative weight).
- **Political orientation — future granularity:** the current questionnaire offers a binary
  `left` / `right` choice (or no-answer). A finer-grained scale (e.g., strong vs. slight lean)
  is a planned future enhancement.
- **Extension architecture unification (outstanding):** the extension currently exists in two
  branches — `feat/extension-scaffold` (TypeScript + esbuild) and `view_interventions` (plain
  .js, no bundler). These must be unified to a single TypeScript + esbuild architecture before
  final delivery.
